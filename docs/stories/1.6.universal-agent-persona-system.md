# Story 1.6: Universal Agent Persona System

## Status

Approved

## Story

**As a** system architect,
**I want** a universal agent system that loads persona specifications from YAML files,
**so that** agent behaviors can be configured without code changes and new personas can be created through configuration rather than development.

## Acceptance Criteria

1. **UniversalAgent Implementation**: `UniversalAgent` class implemented extending `SecureAgent` that can embody any persona
2. **YAML Persona Format**: Persona specification format defined with identity, behavioral_contract, command_interface, resource_dependencies, and knowledge_context sections
3. **Persona Loader**: System loads personas with override precedence (src/agents/personas/ > .bmad-core/)
4. **Initial Persona Specs**: Core persona specifications created (dev.yaml, orchestrator.yaml, release.yaml)
5. **Factory Integration**: Agent factory updated to support persona-based agent creation with backward compatibility
6. **CLI Support**: CLI commands updated to accept persona parameters (--persona flag)
7. **Comprehensive Testing**: Unit tests for persona loading, agent behavior consistency, and error handling

## Dev Notes

### Technical Implementation Details

**Persona Specification Format:**

```yaml
identity:
  name: "Alex"
  id: "dev"
  title: "Orchestra Developer"
  role: "Expert Python/Temporal developer"
  icon: "💻"
  when_to_use: "Code implementation, debugging, refactoring"
  style: "Concise, pragmatic, security-conscious"
  focus: "Executing story tasks with precision"

# Orchestrator persona example:
identity:
  name: "Brendan"
  id: "orchestrator"
  title: "Orchestra Orchestrator"
  role: "Master planning and coordination agent"
  icon: "🎼"
  when_to_use: "Request analysis, planning, workflow coordination"
  style: "Strategic, analytical, decisive, collaborative"
  focus: "Analyzing requests and coordinating development workflows"

behavioral_contract:
  core_principles:
    - "All external API calls must be wrapped in Temporal activities"
    - "Include correlation IDs for tracing"
  interaction_style: "numbered_lists"
  halt_conditions:
    - "Security validation failures"
    - "Ambiguous requirements"

command_interface:
  execution_model: "sequential"
  commands:
    implement-story:
      description: "Implement story following Orchestra workflow"
      execution_pattern: "read_story → implement → test → validate"

resource_dependencies:
  knowledge_sources:
    - "docs/architecture/coding-standards.md"
  tasks:
    - "implement-orchestra-story.md"
  tools:
    - "github-tools"
```

**Key Integration Points:**

- UniversalAgent must maintain all SecureAgent functionality
- Persona loading must be fast (<100ms) for CLI responsiveness
- Error messages must be clear for YAML syntax issues
- Backward compatibility critical for existing workflows

### Security Considerations

- Validate all YAML input to prevent injection attacks
- Sanitize persona specifications before loading
- Ensure persona files cannot execute arbitrary code
- Validate tool references against allowed tool registry

### Testing Strategy

- Unit tests for each component (loader, specs, universal agent)
- Integration tests with real persona files
- Performance tests for persona loading speed
- Error handling tests for malformed YAML
- CLI integration tests with persona parameters
