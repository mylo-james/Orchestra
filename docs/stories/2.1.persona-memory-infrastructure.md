# Story 2.1: Persona Memory Infrastructure

## Status
Ready for Done

## Story
**As a** Product Owner,
**I want** memory infrastructure with sub-workflows that personas can invoke for persistent context and pattern storage,
**so that** personas can automatically capture and retrieve relevant context without additional coordination overhead.

## Acceptance Criteria
1. Memory storage system implemented with namespaced collections per project
2. Memory upsert sub-workflow captures context and patterns from persona executions
3. Memory retrieval sub-workflow provides relevant historical context to personas
4. Memory management sub-workflow handles retention policies and archival
5. Integration with existing Temporal workflow orchestration
6. Integration with existing Qdrant vector database and PostgreSQL metadata storage
7. **Context extraction algorithms achieve >80% relevance score for captured patterns**
8. **Memory retrieval response time <200ms for 95% of queries**
9. **Memory footprint stays within 4GB constraint with automatic cleanup triggers**
10. **Retention policies automatically archive memories older than 90 days**

## Dev Notes

### Architecture Context
- **Database Integration**: Extends existing Qdrant vector database and PostgreSQL metadata storage [Source: architecture/data-and-isolation.md#qdrant]
- **Project Isolation**: Uses namespaced storage per project: orchestra_memory_{project_id} [Source: architecture/data-and-isolation.md#qdrant]
- **Schema Design**: PostgreSQL schema per project for metadata/state with retention/archival policy [Source: architecture/data-and-isolation.md#postgresql]

### Technical Implementation
- **Temporal Workflow Integration**: Leverages existing Temporal orchestration infrastructure [Source: architecture/technical-summary.md#temporal-integration]
- **Sub-workflow Pattern**: Memory operations as composable Temporal sub-workflows
- **Memory Models**: Use Pydantic for data validation following existing patterns [Source: architecture/technical-summary.md#pydantic]
- **Local Deployment**: Must maintain laptop-friendly resource usage <= 4GB memory footprint [Source: requirements.md#NFR2]

- Storage patterns from Story 1.8 provide namespace implementation guidance

### Data Models
- **MemoryRecord**: Core memory storage with embeddings, metadata, confidence scores
- **ContextPattern**: Project-specific interaction patterns with success metrics
- **MemoryIndex**: Fast retrieval indexing with relevance scoring
- **RetentionPolicy**: Configurable memory lifecycle management
